Building a Multi-Market Personalized Offers Engine

Abhilash Ganji January 1, 2025 (Updated April 10, 2026) 18 min read

Most recommendation system tutorials end at model.fit(). The real challenge starts after — when you need to serve personalized predictions to thousands of loyalty users across multiple countries, handle cold-start offers that appear weekly, run rigorous A/B tests against control groups, and give the ops team a dashboard to manage it all.

This post walks through how I designed and deployed a personalized offers recommendation engine for a global QSR (Quick Service Restaurant) chain — from the theory behind hybrid recommenders to the Snowflake-native architecture that runs it in production across multiple markets.

The Problem

Loyalty programs in QSR generate a constant stream of promotional offers: discounts, bundle deals, limited-time items. The challenge is which offers to show which users. Random assignment wastes impressions. Popularity-based ranking ignores individual taste. We needed a system that learns each user's preferences from their redemption history and personalizes the offer set per user, per market.

Key constraints:

  • Multi-market: Each country has different offer catalogs, menu items, and user behavior patterns.
  • Frequent cold start: New offers appear every 1–2 weeks. The model must handle items with zero interaction history.
  • Measurable lift: Every assignment cycle needs A/B evaluation — test group (personalized) vs. control group (random/popularity).
  • Ops-friendly: Non-technical stakeholders need to upload new offers, toggle flags, and view results without touching code.

Why LightFM: The Hybrid Approach

The Three Recommendation Paradigms

Before diving into LightFM, here's a quick lay of the land:

  • Collaborative Filtering (CF): "Users who redeemed similar offers in the past will redeem similar offers in the future." Works great with sufficient interaction data, but fails on new users and new items.
  • Content-Based Filtering: Uses item features (protein type, discount tier, menu category) to recommend similar items. No cold-start for items, but creates filter bubbles.
  • Hybrid: Combines collaborative signals with content features. Best of both worlds — and what we needed.

LightFM's Key Insight

LightFM represents each user and item as the sum of their feature embeddings rather than a single ID embedding. The prediction for user u and item i is: 

score(u, i) = sigmoid( q_u · p_i + b_u + b_i )

where:
  q_u = sum of user u's feature embeddings
  p_i = sum of item i's feature embeddings
  b_u, b_i = user and item biases

This means a brand-new offer with features [BURGER, DISCOUNT_30, COMBO] immediately gets a meaningful embedding — the sum of the learned embeddings for each of those features. No interaction history required.

We trained with WARP (Weighted Approximate-Rank Pairwise) loss, which directly optimizes for top-K ranking quality. Unlike BPR or logistic loss, WARP focuses gradient updates on items that violate the desired ranking — making it ideal when you only care about the top 3 recommendations per user.

The entire pipeline runs natively on Snowflake — no external orchestrator, no separate model server. Here's each layer:

1. Feature Engineering on Snowflake Feature Store

We used Snowflake's ML Feature Store to manage versioned feature views for both users and offers:

  • User features: Derived from loyalty transaction history — frequency of visits, average spend, preferred daypart, cuisine preferences. Refreshed incrementally.
  • Offer features: Categorical attributes from the offer catalog — menu size, protein type, additional products, discount type. One-hot encoded at training time.
  • Interactions: Implicit feedback matrix built from offer redemptions. A user redeeming an offer = positive signal. We used configurable training windows (e.g., data from the last 6–12 months).

Feature views are versioned per country, enabling independent iteration per market without cross-contamination.

2. Per-Market Configuration

Each market has its own YAML config file defining model hyperparameters and feature sets: 

# Example market config
model:
  training_data_start_date: "2025-01-01"
  offer_categorical_features:
    - MENU_SIZE
    - ADDITIONAL_PRODUCT
    - PROTEIN
  number_of_predictions: 3
  model_name: lightfm

lightfm_model:
  item_alpha: 5e-4
  user_alpha: 1e-5
  no_components: 128
  epochs: 8
  num_threads: 4

This separation means adding a new country is just a new YAML file and deploying market-specific Snowflake infra — the training and prediction code stays identical.

3. Training Pipeline

Training runs in Snowpark-powered notebooks that:

  1. Pull user features, offer features, and interactions from the Feature Store.
  2. Build the LightFM Dataset with both user and item features.
  3. Handle cold-start items — offers with no interactions get their identity features set to the average of interacted items' features.
  4. Train the LightFM model with WARP loss and the market's hyperparameters.
  5. Serialize and upload the model artifact to a Snowflake internal stage.

4. Prediction and Assignment

The prediction pipeline loads the trained model, scores all active offers for each loyalty user, and writes the top-N personalized recommendations to an assignment table. Users are split into:

  • TEST group: Gets personalized offer assignments from the model.
  • CONTROL group: Gets random or popularity-based assignments.

This test/control split is maintained per assignment cycle, enabling clean causal measurement of the model's impact.

5. Streamlit Ops Dashboard

The ops team uses a Streamlit app (deployed on Snowflake) to:

  • Upload offers: Import new offer data from Excel, with validation on required columns and data types.
  • Manage flags: Toggle which offers are in the full assignment list and which are reserved for the control group.
  • View results: Interactive charts showing redemption rates, potential sales increase, and test-vs-control performance per assignment cycle.

Handling Cold Start in Practice

Cold start isn't a one-time problem — it's a recurring one. New offers launch every 1–2 weeks. Here's how we handled it:

  • New offers: Since LightFM uses feature-sum embeddings, a new offer immediately gets a representation from its categorical features (protein type, discount type, etc.). We also implemented a fallback: items with zero interactions get their identity features set to the average identity features of items that do have interactions. This prevents new items from getting degenerate zero-vector representations.
  • New users: User features (visit frequency, spend patterns, daypart preferences) provide a baseline embedding even before any offer interactions.
  • New markets: When launching in a new country with zero data, we use popularity-based fallbacks and transition to personalized recommendations as interaction data accumulates.

Every assignment cycle, the dashboard shows:

  • Potential sales increase: Estimated revenue uplift from personalized assignments.
  • Transaction performance: Pre-vs-post comparison of average transaction value for users receiving personalized offers.
"The absence of a click doesn't mean the user dislikes the item — it might mean they never saw it. Treating all missing interactions as negative examples is the most common mistake in production recommender systems."

Implicit Feedback and Negative Sampling

We don't have explicit ratings — only implicit signals: offer redemptions. This changes model training fundamentally. A user not redeeming an offer could mean they weren't interested, or it could mean they never saw it.

LightFM's WARP loss handles this elegantly — it samples negative items during training and focuses updates on violations where a non-interacted item scores higher than an interacted one. The number of sampling attempts before finding a violation is itself informative: if it takes many samples, the model is already ranking well for that user.

Lessons Learned

  1. Start simple, measure, then improve. Our first version was popularity-based assignment. It established the baseline. LightFM beat it consistently, but without that baseline, we'd have no way to quantify the lift.
  2. Feature engineering > model complexity. Adding offer categorical features (protein type, discount tier) improved precision@3 more than doubling the embedding dimension did.
  3. Per-market configs are essential. Germany and Switzerland had different optimal hyperparameters, different feature sets, and different training windows. A one-size-fits-all config would have underperformed in at least one market.
  4. Cold-start handling is a feature, not an afterthought. The identity-feature averaging trick for new offers was critical — without it, weekly new offers would get zero scores and never surface.
  5. Give ops the tools. The Streamlit dashboard reduced the feedback loop from "file a ticket and wait" to "upload, flag, and see results" — which meant faster iteration on the offer catalog itself.
  6. A/B test every cycle. We maintained test/control splits in every assignment. Two benefits: rigorous measurement of model lift, and a built-in fallback if the model ever degrades.

Tech Stack Summary

  • Model: LightFM with WARP loss, 128-dimensional embeddings
  • Data platform: Snowflake (Feature Store, Internal Stages, Snowpark)
  • Training: Snowpark-powered Jupyter notebooks with per-market YAML configs
  • Infrastructure: SQL-based deployments for common and market-specific infra
  • Dashboard: Streamlit on Snowflake with Altair visualizations
  • Languages: Python 3.11, SQL
  • Libraries: LightFM, Pandas, SciPy, Pydantic, Altair, Snowpark ML

What's Next

The current system works well for the scale we're at. Future directions:

  • Sequential models: Transformer-based approaches (SASRec, BERT4Rec) to capture temporal patterns in redemption sequences.
  • Real-time re-ranking: Context-dependent signals (time of day, day of week, weather) applied at serving time.
  • More markets: The per-market config pattern makes expansion straightforward — the bottleneck is data accumulation, not engineering.

The key takeaway: a well-engineered hybrid model with proper feature engineering, cold-start handling, and rigorous A/B testing will outperform a fancy deep learning model deployed without this infrastructure. The system around the model matters more than the model itself.
