Building Recommendation Systems That Actually Scale

Abhilash Ganji January 1, 2025 15 min read

Recommendation systems power some of the most impactful features in modern products — from Netflix's "For You" row to Amazon's "Customers who bought this also bought." But building a recommender that works in a notebook is trivially different from one that scales to millions of users in production.

In this post, I'll walk through the major recommendation paradigms, share the architecture that powered our restaurant-scale personalized offer engine at EPAM, and cover the hard lessons I learned deploying recommenders that actually survive real traffic.

The Three Paradigms

1. Collaborative Filtering

Collaborative filtering is the workhorse of recommendation systems. The core idea: users who agreed in the past will agree in the future. There are two main approaches:

  • User-based CF: Find users similar to the target user and recommend what they liked. Simple but doesn't scale — computing user-user similarity across millions of users is expensive.
  • Item-based CF: Find items similar to what the user already liked. More stable and cacheable. Amazon popularized this approach in their 2003 paper.

The fundamental limitation of CF is the cold start problem. A new user with no interaction history gets no signal. A new item with no ratings gets no recommendations. This is where content-based approaches help.

2. Content-Based Filtering

Content-based systems recommend items similar to what the user has liked before, using item features rather than community behavior. For restaurant offers, this means matching cuisine preferences, price sensitivity, location patterns, and time-of-day habits.

The advantage: no cold start for items (we know item features from day one). The disadvantage: limited discovery — users get trapped in a "filter bubble" of things they already like.

3. Hybrid Approaches

Production systems almost always use hybrid approaches. At EPAM, we combined collaborative signals with content features using LightFM, a hybrid matrix factorization model that handles both interaction data and side features natively.

Matrix Factorization: The Foundation

Matrix factorization decomposes the user-item interaction matrix into two lower-dimensional matrices — user embeddings and item embeddings. The dot product of these embeddings predicts the interaction score: 

R ≈ U × V^T

where:
  R = user-item interaction matrix (m × n)
  U = user latent factors (m × k)
  V = item latent factors (n × k)
  k = embedding dimension (typically 32-128)

LightFM extends this by incorporating feature embeddings. Each user and item is represented not just by their ID embedding, but by the sum of their feature embeddings. This is what solves cold start — a new user with known demographics gets a meaningful embedding from day one.

Our Production Architecture at EPAM

Here's the architecture we deployed for personalized restaurant offer recommendations, serving predictions to thousands of customers daily:

  1. Data Layer (Snowflake): User interaction events (clicks, redemptions, ignores), user profiles, and offer metadata stored in Snowflake with incremental refresh.
  2. Training Pipeline: LightFM model trained on implicit feedback (offer clicks and redemptions). We used WARP loss for ranking optimization, which directly optimizes for top-K recommendation quality.
  3. Model Registry (MLflow): Versioned models with A/B test metadata. Every model gets evaluated against a holdout set before promotion.
  4. Serving Layer (FastAPI + Docker): Pre-computed recommendations cached in Redis for sub-10ms latency. Real-time re-ranking for context-dependent features (time of day, location).

Handling Implicit Feedback

In most real-world systems, you don't have explicit ratings — you have implicit feedback: clicks, views, purchases, time spent. This changes everything about how you train the model.

"The absence of a click doesn't mean the user dislikes the item — it might mean they never saw it. Treating all missing interactions as negative examples is the most common mistake in production recommender systems."

We handled this with negative sampling — for each positive interaction, we sampled a fixed number of negative items the user hadn't interacted with, weighted by item popularity. This gives the model a realistic view of what "not interested" looks like.

The Cold Start Problem in Practice

Cold start isn't one problem — it's three:

  • New user cold start: We used feature-based embeddings (age group, location, signup source) to bootstrap new user representations. After 3-5 interactions, the collaborative signal takes over.
  • New item cold start: Offer metadata (cuisine type, discount percentage, restaurant category) provides immediate item embeddings. New offers get a "exploration bonus" in the re-ranking layer.
  • System cold start: When launching in a new market with zero data, we used popularity-based fallbacks and gradually transitioned to personalized recommendations as interaction data accumulated.

Evaluation: Beyond Accuracy Metrics

Offline metrics like precision@K, recall@K, and NDCG tell you if the model is learning. But they don't tell you if users are happy. We tracked:

  • Offer redemption rate: The ultimate business metric — did the user actually use the recommended offer?
  • Diversity: Are we recommending variety or the same 5 restaurants? We measured intra-list diversity using genre entropy.
  • Coverage: What percentage of total offers appear in at least one user's recommendations? Low coverage means long-tail items are invisible.
  • Freshness: Are we surfacing new offers or just recommending established ones?

Lessons Learned

  1. Start simple. Our first version was item-based CF with popularity fallback. It beat "show random offers" by 8%. We added LightFM later and gained another 4%. Don't architect for deep learning day one.
  2. Feature engineering matters more than model choice. Adding "days since last visit" as a user feature improved redemption rate more than switching from ALS to LightFM did.
  3. Cache aggressively. Pre-compute top-50 recommendations per user nightly. Re-rank on-the-fly for context. This kept P99 latency under 15ms.
  4. Monitor for feedback loops. Recommenders create their own training data. If you only show popular items, you only get signal on popular items. We added exploration (10% random) to break the loop.
  5. A/B test everything. What looks good in offline metrics can underperform online. We ran every model change through a 2-week A/B test before promotion.

What's Next: Deep Learning RecSys

Matrix factorization works remarkably well for its simplicity, but the field is moving toward deep learning approaches — two-tower models, transformer-based sequential recommendation, and graph neural networks for capturing complex user-item-context relationships.

The key question isn't "is deep learning better?" — it's "do you have enough data and complexity to justify the infrastructure cost?" For most mid-scale systems, LightFM or a well-tuned ALS model will outperform a poorly-tuned deep model every time.
